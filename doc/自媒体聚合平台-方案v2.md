# 自媒体创作者聚合平台 — 方案 v2（Demo优先 · 聚合核心）

## Context（背景与目标）

客户想做一个面向**高端自媒体创作者**的 SaaS 聚合平台，核心四大功能：多平台文案创作 / 爆款拆解 / AI视频重生成 / 一键多平台发布。

**v2 方案的核心转变（基于讨论对齐）：**
- ❌ 不再一上来铺完整架构 → ✅ **第一阶段先做纯前端高保真原型**（huashu 设计风格，mock 数据，部署阿里云给客户看）
- ❌ 不依赖爬虫（合规风险高）→ ✅ **数据走付费数据 API**（新榜为主），爬虫降级为补充
- ❌ 不写死某家模型 → ✅ **全部走你的 sub2api 中转站**，去掉 Gemini，模型按中转站支持项动态选
- ❌ 不武断推荐框架 → ✅ **多Agent框架给客观对比表，你来拍板**；demo 阶段先不上多Agent
- ✅ 补全：服务器规格、网络环境、成本估算、前端风格（huashu）

**最终目标**：先用 demo 拿到客户认可 → 再聚焦"聚合"这一真正难点逐步扩展功能。

## 一、阶段化路线（分步骤实现，先 demo 后扩展）

| 阶段 | 周期 | 目标 | 是否接AI | 交付物 |
|------|------|------|---------|--------|
| **Step 1：高保真原型** | 1-2周 | 给客户看，拿反馈 | ❌ mock | huashu风格前端，部署阿里云 |
| **Step 2：数据聚合层** | 2-3周 | 打通付费API，跑通"拆解"数据来源 | 部分 | 新榜API接入 + 统一数据模型 |
| **Step 3：AI创作最小版** | 2-3周 | 文案创作真跑通（sub2api） | ✅ | 单LLM + function calling，**不上多Agent** |

> **关键原则**：demo → 数据 → 创作 → 发布 → 视频 → 多Agent。难度递增、价值递增、风险后置。**多Agent是优化手段，不是起步必需**。

## 二、Step 1 详解：高保真原型（马上做，给客户看）

### 技术栈
- **前端**：Next.js 15 + TypeScript + Tailwind + Shadcn/UI + Framer Motion（动效）
- **设计语言**：**复用 huashu-design 的视觉体系**（`projects/huashu-design/assets` 里的 design_canvas、brand spec、动画组件）
- **数据**：全部 mock（写死 JSON），不接后端
- **部署**：阿里云 ECS + Nginx，或直接 Vercel（最快）

### 页面规划（5个核心页面）
1. **首页/Landing** — 产品价值主张 + huashu 风格 hero 动画
2. **创作工作台** — 选平台（抖音/小红书/公众号/知乎/B站）→ 输入主题 → 展示生成的多平台文案（mock）
3. **爆款拆解** — 粘贴链接 → 展示拆解卡片（钩子/结构/情绪曲线/数据，mock）
4. **视频重生成** — 上传素材区 + 生成进度 + 成品预览（mock 视频/截图）
5. **发布中心** — 多平台勾选 + 一键发布按钮 + 发布结果（mock）

### 为什么纯前端够用
- 高端客户买单的是**产品 vision 和体验**，不是"AI真的在跑"
- 原型能验证：交互是否顺畅、功能是否完整、设计是否高级
- 拿到反馈再决定哪些功能真做，避免过度开发

### 验证标准
- 客户能在 demo 上完整走通"拆解→创作→发布"三个流程的视觉体验
- 设计风格达到"像真团队做的"（huashu 标准）
- 部署后可通过公网 URL 访问

## 三、聚合层设计（核心难点 · Step 2）

> 这是整个产品技术上的真正难点：**怎么获取数据、怎么标准化聚合、怎么支撑多场景复用**。

### 数据源策略（付费API为主）
| 数据源 | 用途 | 覆盖 | 备注 |
|--------|------|------|------|
| **新榜有数 API**（主） | 热点/榜单/达人/内容数据 | 微信/抖音/快手/小红书/B站/视频号 | 覆盖最广，有成熟API开放服务，**首选接洽商务** |
| **蝉妈妈**（补充） | 抖音电商/带货数据 | 抖音为主 | 电商场景强，价格偏高 |
| **灰豚数据**（补充） | 性价比多平台 | 抖音/小红书 | 综合性价比 |
| **爬虫**（降级） | API覆盖不到的长尾/详情 | 按需 | MediaCrawler/feedgrab 备用，仅补充 |

### 核心设计：统一数据模型（聚合的关键）
不管数据来自新榜还是蝉妈妈，都**归一化成一套内部 schema**：
```
ContentItem {
  platform, content_id, author, title, text,
  media[],  // 视频/图片/音频统一抽象
  metrics {  // 统一指标（各平台字段名不同，这里抹平）
    views, likes, comments, shares, collects, engagement_rate
  },
  published_at, tags, category,
  raw_source  // 保留原始来源用于追溯
}
```
**收益**：上层 Agent/前端只面对统一模型，新增数据源 = 加一个 adapter，不动业务逻辑。

### 聚合服务架构
```
[前端/Agent] → [聚合API（统一schema）] → [数据源适配器层]
                                         ├─ 新榜 adapter
                                         ├─ 蝉妈妈 adapter
                                         ├─ 灰豚 adapter
                                         └─ 爬虫 adapter（降级）
                    ↓
              [Redis缓存层]（热点数据缓存，降成本）
```

### 成本控制
- 付费API按调用计费 → **必须做缓存**（热点榜单缓存1小时，详情缓存24小时）
- 按租户配额限流（new-api / sub2api 已有计量能力可复用思路）

## 四、模型策略（全走 sub2api，去掉 Gemini）

**核心原则**：所有 LLM 调用走你的 **sub2api 中转站**，不直连各家。模型选型按中转站实际支持的来定，不写死。

| 任务 | 倾向模型（待确认中转站支持） | 备注 |
|------|---------------------------|------|
| 文案创作（中文） | Claude / GLM-5.x | 中文创作能力强 |
| 爆款逻辑分析 | Claude（深度推理） | 拆解因果链 |
| 视觉理解（替代Gemini） | GLM-4.6V / Qwen-VL / 中转站支持的视觉模型 | ❌ 去掉Gemini |
| 视频生成 | **Seedance 2.0**（字节，主）+ 即梦（补充） | 走火山引擎API |
| 质量评估Judge | Claude | 当裁判 |

> **模型这块你说了"后面再讨论"**，所以 Step 1-2 不涉及，等接 AI（Step 3）时再根据 sub2api 实际能力定。**先确认你中转站支持哪些模型**，是这步的前提。

## 五、多Agent框架客观对比表（你来选，不武断）

> **重要前提**：Step 1-4 根本不需要多Agent——单 LLM + function calling 就能跑通文案创作和拆解。**多Agent是 Step 6 才考虑的优化**，等功能复杂、出现明显编排痛点时再上。

### 6 个方案横向对比

| 框架 | 核心思想 | 优势 | 劣势 | 适合本产品场景 |
|------|---------|------|------|--------------|
| **单LLM + function calling（自研）** | 一个模型+工具调用，自己写流程 | 最简单、零依赖、完全可控、调试容易 | 流程复杂后难维护、无并发编排 | ✅ **Step 1-4 起步首选** |
| **LangGraph** | 图状态机 | 可视化、时间旅行调试、human-in-the-loop、生产级 | 学习曲线陡、样板代码多 | ✅ 适合有状态创作流水线（Step 6候选） |
| **CrewAI** | 角色协作（雇佣团队） | 原型最快、YAML配置、直观 | 无沙箱、生产级弱、Python-only | ⚠️ 快速验证多Agent效果时 |
| **OpenAI Agents SDK** | handoff模型 | handoff最干净、微服务式、分布式友好 | 绑OpenAI生态（走中转站可缓解） | ⚠️ 若强调服务化分发 |
| **Claude Agent SDK** | Claude深度集成 | 轻量、与Claude/Claude Code生态打通 | 绑Claude | ⚠️ Claude中心化场景 |
| **AutoGen** | 对话式多Agent | 灵活、研究导向、Docker沙箱 | v0.4破坏性更新、文档滞后 | ❌ 偏研究，生产不推荐 |

### 我的倾向（仅供参考，你拍板）
- **Step 1-4**：单 LLM + function calling（不上框架）
- **Step 6 若需多Agent**：**LangGraph**（有状态、可观测、human-in-the-loop 对高端产品"人工审稿"刚需）
- 但**不锁定**——Step 6 时会基于真实痛点重新评估，CrewAI（快）和 OpenAI Agents SDK（干净）都在候选

> 这块我可以后面跟你详细展开任何一个的细节。

## 六、服务器 / 网络 / 成本（补全）

### 阿里云 ECS 实际规格（已升级，2026-06-15 核实）
- **当前配置**：4核 / **7.3G RAM** / 40G磁盘（**已用72%，仅剩11G**）/ Alibaba Cloud Linux 3
- **SSH 别名**：`my-server`（~/.ssh/config），IP `47.100.247.237`，User `root`
- **已装环境**：✅ nginx 1.26 + Node v20.20 + pm2（开箱即用）
- **占用情况**：80/443 跑 nginx；**3000端口已被现有 node 进程占用**（部署需端口/路径隔离）

| 用途 | 当前是否够用 | 说明 |
|------|------------|------|
| **Step 1 原型** | ✅ 完全够 | 纯前端静态文件，nginx 托管 |
| **Step 2-4 后端** | ⚠️ 勉强够 | 7.3G够跑 FastAPI+Redis+PG，但**磁盘只剩11G要先清理** |
| **Step 5 视频** | ❌ 建议升级到 8核16G | 视频处理吃CPU；**磁盘需扩容**（视频素材占空间大） |
| **存储** | OSS对象存储 | 视频素材别放本地盘 |

> ⚠️ **两个部署注意点**：① 磁盘仅剩11G，部署前先 `df -h` + 清理；② 3000端口被占，新项目用其他端口（如3001）或走nginx反代。

### 网络环境要求
| 场景 | 要求 |
|------|------|
| 调用付费数据API（新榜等） | 国内直连即可 |
| 调用大模型API（经sub2api中转） | **取决于中转站部署位置**——若中转站在国内/有国内入口则直连，否则需确认网络 |
| 调用Seedance/即梦（火山引擎） | 国内直连 |
| 爬虫降级（若用） | 可能需代理IP池，但已降为补充，优先不用 |
| RPA发布（bb-browser） | 国内直连，复用真实登录态 |

> **网络是潜在坑**：sub2api 中转站部署在哪、模型实际从哪出，直接决定要不要配代理。**这块需要你告诉我中转站现状**。
